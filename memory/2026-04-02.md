- Operational lesson from today's debugging: restarting or reloading `openclaw-gateway` can temporarily interrupt Feishu message delivery/dispatch because browser control, agent dispatch, config loading, and chat transport are coupled to the same gateway process. Treat gateway restart/reload as a high-risk action.
- New rule for future work: before any action likely to restart/reload `openclaw-gateway` (for example `systemctl --user restart openclaw-gateway.service`, gateway config apply/patch that triggers restart, or other gateway-level reloads), explicitly warn the user that messaging may be briefly interrupted, estimate the outage window, then perform a local post-restart self-check (`systemctl --user status`, recent logs, plugin registration) before continuing other debugging.
- Preferred workflow going forward: maximize read-only diagnostics first; batch necessary gateway-impacting changes into as few restarts as possible; separate browser-instance debugging from gateway restarts whenever possible.
