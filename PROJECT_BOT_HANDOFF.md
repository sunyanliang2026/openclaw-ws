# 主助理 ↔ project-bot 协作规则

这份文件定义：主助理什么时候把任务交给 project-bot，以及交接时应携带哪些信息。

## 目标
- 老板只需要跟主助理说一次
- 主助理负责拆解任务
- 工程执行部分交给 project-bot
- 最终仍由主助理统一汇报结果

## 默认入口
老板默认只找主助理。

除非老板明确要求直接找 project-bot，否则：
- 主助理是总入口
- project-bot 是后台执行器

## 主助理应转交给 project-bot 的任务类型
以下任务默认优先考虑交给 project-bot：

1. GitHub 仓库收编
   - fork
   - clone
   - 配置 `origin` / `upstream`

2. 项目复刻
   - 拉取开源项目
   - 建立本地工作目录
   - 跑通项目

3. 工程执行任务
   - 安装依赖
   - 启动项目
   - 调试报错
   - 修复构建问题

4. 多文件代码改动
   - 功能实现
   - 批量重构
   - 结构性调整

5. 仓库维护任务
   - 同步 upstream
   - 处理分支
   - 准备提交内容

## 主助理不应转交的任务类型
以下任务默认由主助理自己处理：
- 日常对话
- 需求澄清
- 文档整理
- 方案比较与决策建议
- 提醒、备忘、轻量信息查询
- 纯只读的小范围查看

## 转交时必须明确的信息
当主助理把任务交给 project-bot 时，至少应携带：

1. 任务目标
   - 最终想达成什么

2. 目标仓库 / 资源
   - URL
   - 项目名
   - 目标目录

3. 允许范围
   - 是否允许 fork
   - 是否允许 clone
   - 是否允许改代码
   - 是否允许 commit
   - 是否允许 push
   - 是否允许删除文件

4. 交付要求
   - 需要回报什么
   - 需要哪些结果
   - 卡住时应如何汇报

## project-bot 返回结果的默认格式
project-bot 应尽量返回以下结构：

1. 完成状态
   - 已完成 / 部分完成 / 被阻塞

2. 已执行动作
   - 做了哪些步骤

3. 变更范围
   - 改了哪些文件
   - 建了哪些目录 / 分支 / 仓库

4. 当前阻塞点
   - 卡在哪里
   - 需要老板做什么

5. 下一步建议
   - 推荐接下来做什么

## 默认边界
### 主助理
- 负责理解任务
- 负责拆解任务
- 负责结果汇总
- 负责对老板回复

### project-bot
- 负责工程执行
- 不作为老板的默认入口
- 不擅自扩展任务范围
- 不擅自对外发布内容

## 默认原则
- 老板不需要重复说两遍
- 没有明确授权，不自动 commit / push / delete
- 优先把边界说清楚，再开始执行
- 工程细节由 project-bot 处理，结果由主助理收口
